## Primitive 1 — p3 realization datum

A `p3 realization datum` is a record of the minimal objects required before any `p = 3` comparison can be stated.

It has fields:

```text
D_3 = (Source_3, gamma_3, Target_3, phase_3, output_3)
```

where:

1. `Source_3` is the selected cyclic source for the proposed `p = 3` comparison;
2. `gamma_3` is a selected generator of that cyclic source;
3. `Target_3` is the finite phase target `mu_3`;
4. `phase_3` is the assigned value of `gamma_3` in `mu_3`;
5. `output_3` is the finite value read from the proposed realization.

A `p3 realization datum` is only a record shape. It does not assert that such data have been constructed for any proof family, gate fixture, or dynamical system.

## Primitive 2 — mu_3 finite-output map

A `mu_3 finite-output map` is a map

```text
Omega_3: Source_3 -> mu_3
```

from the selected cyclic source into the third roots of unity.

The map is determined by the image of the selected generator:

```text
Omega_3(gamma_3) in {1, omega, omega^2}.
```

If `Source_3` is cyclic of order 3, then a nontrivial `mu_3` output means

```text
Omega_3(gamma_3) = omega
```

or

```text
Omega_3(gamma_3) = omega^2.
```

This definition does not construct `Source_3`. It only states the finite target and evaluation shape once a source is specified.

## Primitive 3 — Eisenstein phase target

The `Eisenstein phase target` is the finite target `mu_3` viewed through the Eisenstein source capture.

It is recorded as:

```text
omega = exp(2*pi*i/3)
mu_3 = {1, omega, omega^2}
Z[omega] = {a + b omega : a,b in Z}
```

The target is `mu_3`, not the full Eisenstein unit group `mu_6`, unless a later PR explicitly changes convention.

The Eisenstein norm

```text
N(a + b omega) = a^2 - a*b + b^2
```

is available as arithmetic support. The norm is not itself a phase comparison theorem.

## Primitive 4 — p3 admissibility conditions

A proposed `p = 3` realization is `p3-admissible` only if all of the following are supplied:

1. a cyclic source `Source_3` with selected generator `gamma_3`;
2. a specified map `Omega_3: Source_3 -> mu_3`;
3. a selected nontrivial target value `omega` or `omega^2` for `Omega_3(gamma_3)`;
4. a source-side construction explaining why `Source_3` is the correct source for the proposed comparison;
5. a proof-family or fixture context explaining what the finite output is meant to measure;
6. an explicit statement of whether the realization is merely vocabulary, a conjectural scaffold, a tested fixture, or a theorem-grade construction.

Without these six items, a `p = 3` comparison claim is not admissible in this repository.

## Primitive 5 — p3 comparison boundary / nonclaim envelope

The `p3 comparison boundary / nonclaim envelope` is part of the scaffold itself.

Any PR using this vocabulary must preserve the following boundary unless it explicitly opens and justifies a separate theorem-state update:

1. no generic `p` theorem;
2. no closed `p = 3` fixture;
3. no odd-prime dynamical target;
4. no general encoding theorem;
5. no Hodge strengthening;

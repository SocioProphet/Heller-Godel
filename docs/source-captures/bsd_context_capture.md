## Role in Heller-Godel

The BSD program material is context only. It is not a reproducibility dependency for Heller-Godel and does not bear theorem weight for proof-character claims.

Its value is methodological:

- evidence taxonomy;
- workstream lane organization;
- strict promotion gates;
- manifest discipline;
- explicit witness handling;
- failed-engine rollback rather than claim laundering;
- milestone-level audit reports.

## Evidence taxonomy pattern

The BSD lane separated:

```text
unconditional theorem-grade rows
conditional or incomplete rows
experimental engines
validation failures
promotion candidates
```

Heller-Godel imports the discipline, not the mathematics. The analogous Heller-Godel statuses are:

```text
definition
theorem-established
computed-identity
construction-fixed-scope
conjecture
future-horizon
false-as-stated
corrected
```

## Rollback precedent

The BSD M3 snapshot is useful because it records an engine failure and rollback instead of promoting invalid numerical results. Heller-Godel should follow the same pattern:

```text
if a computation fails validation, preserve it for review but do not promote it into theorem claims.
```

This is directly relevant to the `zeta_p` correction. The earlier nonabelian central-extension interpretation is preserved as a historical overclaim but is not promoted into the current theorem stack.

## Explicit witness precedent

The BSD 53-row resolution is useful as a model of explicit witness handling: a conditional status became theorem-grade only when exact rational points / triangles were provided and verified.

Heller-Godel analogue:

```text
future-horizon ideas become theorem-core only when the missing object is constructed and verified.
```

For example:

- Chern-class language requires a line bundle/local system, connection, curvature, and base map;
- Moufang holonomy requires proof moduli and a holonomy assignment;
- operator-shell dynamics require an operator family tied to proof-character shell data.
